laravel-authorization-review

● available security IDOR / BOLA access control v0.1.0 MIT

A Laravel-native authorization reviewer. Point it at any Laravel project and it finds broken object-level authorization (IDOR) — the bug class static scanners structurally miss — without ever editing your code.

What it does

Static scanners trace where untrusted input flows. They can't tell you whether Invoice::find($id) should have been scoped to the logged-in user — that's a question about intent, and it's exactly the bug behind IDOR / broken object-level authorization (BOLA), #1 in the OWASP API Security Top 10. This skill is the judgment layer for that gap.

  • ▹
    Walks the authorization chain of every route — authentication → authorization (policy/gate/can:/FormRequest) → object scoping → API Resource output.
  • ▹
    Catches the classic Laravel IDOR — route-model binding (update(Post $post)) resolves any record by id but does not authorize it; a missing authorize() or owner scope is the hole.
  • ▹
    Anchored to ground truth — every finding traces to a real php artisan route:list --json entry and a cited controller file:line. No anchor, no finding.
  • ▹
    Confidence-rated — High (structural), Medium (depends on whether a model is owned), Low (hardening). Medium/Low are phrased as "verify", never asserted.
  • ▹
    False-positive aware — recognizes every legitimate authorization pattern and knows public-by-design routes (login, webhooks with signatures), so it won't cry wolf.
  • ▹
    Per-route coverage map — shows every route × every layer, so you see what's protected and what's checked, not just a list of negatives.
Advise-only. It reads files and runs one read-only command (php artisan route:list --json), then hands you findings + fix sketches. It never edits your routes, controllers, policies, or code — you apply every fix yourself.

Example output

A real review run, condensed. The route inventory and cited lines are hard facts; whether a model is "owned" or a field is sensitive is the skill's judgment — stated as an assumption to confirm.

## 🔴 Verify & fix now
DELETE /api/invoices/{invoice} → InvoiceController@destroy:88
  → implicit binding resolves ANY invoice; no authorize(), no owner scope → IDOR
  → Invoice is owned (invoices.user_id, User::invoices())  [High]
  → fix: $this->authorize('delete', $invoice)  (InvoicePolicy::delete is missing)

## 🟡 Review — needs your judgment
GET /api/invoices/{invoice} → show() returns model, no authorize('view')  [Medium]
  → confirm: owned vs. team-shared → add authorize() or scope to the team

The lanes

🔴 Verify & fix now

High-confidence structural holes — no auth/authz/scope on owned data. Confirm with the cited line, add the check, add an authorization test.

🟡 Review

Likely IDOR or data leak that depends on whether a model is owned or a field is sensitive. Decide the intent, then scope or authorize.

🔵 Hardening

Defense-in-depth — missing policy, mass-assignment of ownership keys. Tidy when convenient.

Install & use

You don't need the whole library or a marketplace — drop just this one skill folder into your project's .claude/skills/:

your Laravel project
# grab just this skill (per-project)
npx degit ArtemProshkovskiy/laravel-maintenance-skills/maintenance/skills/laravel-authorization-review \
  .claude/skills/laravel-authorization-review

Use ~/.claude/skills/ instead to install it globally (every project). No npx? Clone the repo and copy the maintenance/skills/laravel-authorization-review folder into .claude/skills/. Want every skill at once? Add the plugin marketplace (optional).

Open Claude Code in your project

Start it inside your Laravel project folder (the one with artisan).

Ask in plain language

"review my authorization", "find IDOR in this app", "which routes have no auth?", "review authorization on the routes in this PR" — the skill activates automatically.

Review, then fix

You get a coverage map + confidence-rated findings with evidence. Nothing is changed for you — apply each fix on a branch, with authorization tests.

Boundaries

  • does not edit routes, controllers, policies, or any code — you apply every fix;
  • does not invent a route, policy, or method — only what route:list and your files contain;
  • does not cover Livewire / Filament / Nova action authorization — those live outside the route:list anchor and use their own models (a future version may add them);
  • does not judge business-logic authorization (approval states, time-based rules) — flagged for a human;
  • does not catch IDOR via indirect references not visible in code, or infra/gateway authz.

It is not a pentest and not a security product: a finding-free report means "nothing broken in the layers checked", not "secure".
